## 第一部分：项目定位与竞争优势

### 1.1 市场生态位分析

当前 AI 工具市场大致分为四个象限：

|                | 纯聊天/轻量       | 完整 Agent 工作台      |
|:--------------:|:------------------:|:----------------------:|
| **云端 SaaS**  | ChatGPT, Claude.ai | Dify, Coze, Langflow   |
| **本地优先**   | LM Studio, Jan.ai  | **灵犀（蓝海位置）**   |

**灵犀所处的"本地优先 + 完整 Agent 工作台"象限，目前几乎没有直接竞品。** 这不是偶然——同时做到本地数据主权、多模型无关接入、Agent 间协作网络、可视化工作流、企业 IM 集成、零配置桌面安装，技术栈跨度和工程量极大，构成了天然的进入壁垒。

### 1.2 竞品矩阵

| 维度 | 灵犀 | ChatGPT | Claude.ai | Dify | Coze | LobeChat | AnythingLLM | LM Studio |
|------|:----:|:-------:|:---------:|:----:|:----:|:--------:|:-----------:|:---------:|
| 数据本地化 | **完全本地** | 云端 | 云端 | 云/自部署 | 云端 | 可选 | 本地 | 本地 |

| Agent 协作网络 | **Nexus** | 无 | 无 | 无 | 无 | 无 | 无 | 无 |
| 可视化工作流 | 6种节点 | 无 | 无 | **完整** | **完整** | 无 | 无 | 无 |

**结论：** 灵犀在"本地化 + Agent 协作 + 零配置部署"三个维度上没有直接竞品。Dify/Coze 在工作流编排上更成熟，但都是云端 SaaS 模式，不解决数据主权问题。

### 1.3 六大核心壁垒

**壁垒一：Project Nexus — Agent-to-Agent 自动协作网络**

这是灵犀最独特的差异化能力。局域网 mDNS 自动发现 + PSK 密钥建联 + 双向流式对话 + 广域网信令中继，构成了一个去中心化的 Agent 通信网络。目前市场上没有第二个桌面 AI 工具实现了类似功能。Nexus 不是一个普通功能，而是一个潜在的**网络效应平台**。

**壁垒二：零依赖部署**

内嵌 Go 后端 + Node.js 运行时 + whisper.cpp 语音引擎，双击 .dmg 安装即可使用。不需要 Docker、Python、数据库服务或任何云端依赖。这意味着用户获取成本极低——IT 基础设施为零的个人用户也能使用全部功能。

**壁垒三：数据主权**

SQLite 本地存储 + macOS Keychain / Windows DPAPI 密钥加密。在 AI 监管日益趋严的全球环境下（欧盟 AI Act、中国《生成式人工智能服务管理暂行办法》），数据不出本地是企业和敏感行业的刚需。

**壁垒四：模型无关**

Bridge 路由层（LiteLLM + llm-bridge）自动完成 Anthropic ↔ OpenAI 协议双向翻译，用户可在 14+ 家模型供应商之间自由切换。任何一家供应商涨价、停服或被制裁，用户不受影响。

**壁垒五：全链路闭环**

创建智能体 → 装备技能/知识库/MCP → 执行对话任务 → 编排工作流 → 定时自动化 → 跨实例 Agent 协作。一个工具完成从"创建 AI 助手"到"自动化运营"的完整链路。

**壁垒六：架构可扩展性**

Electron + React + Go 三层清晰分离。纯 Go WASM SQLite 无 CGO 依赖，跨平台构建已验证（macOS arm64 + Windows x64）。后端 70+ API、WebSocket 流式、mDNS 发现、定时调度器等基础设施完备。

## 第二部分：商业化路径全景分析

### 2.1 路径 A：企业私有化部署

**商业模式：** 开源社区版免费 + 企业版按 seat 收费

**定价参考：** 49-99 元/人/月（年付 8 折）

**需要新增的核心功能：**

| 功能 | 描述 | 预估工期 |
|------|------|----------|
| RBAC 多用户权限 | 管理员/员工/访客三级权限，基于现有 users 表扩展 role 字段 | 3-4 周 |
| 审计日志 | audit_logs 表，记录对话/Agent 创建/API Key 变更等操作，支持导出 | 2-3 周 |
| 集中式 API Key 管理 | 管理员统一配置模型密钥，员工无需自行管理 | 1-2 周 |
| LDAP/AD 认证 | 企业级认证集成，替代当前游客 + OAuth SSO | 2-3 周 |
| 部门级配额 | 用量配额按部门/团队分配，预算独立控制 | 2 周 |
| Agent 内部市场 | 企业内共享智能体模板，团队间复用最佳实践 | 2-3 周 |

**目标客户画像：**
- 金融机构（银行风控团队、证券研究部）—— 数据合规刚需
- 律所 / 法务部门 —— 合同审查、法规检索，不能上传到云端
- 医疗机构 —— 病历分析、医学文献检索，隐私要求极高
- 政府 / 国企 —— 国产化 + 数据不出网，刚需本地部署
- 50-500 人中型企业技术/运营团队 —— 需要 AI 自动化但预算有限

**收入测算：**

| 场景 | 客户数 | 平均 seat | 单价 | 月收入 | 年收入 |
|------|--------|-----------|------|--------|--------|
| 起步期 | 3 家 | 30 人 | 69 元 | 6,210 | 74,520 |
| 成长期 | 15 家 | 50 人 | 69 元 | 51,750 | 621,000 |
| 规模期 | 50 家 | 80 人 | 69 元 | 276,000 | 3,312,000 |

**优势：**
- 变现速度最快，企业客户付费意愿明确
- 单客户价值高，客户粘性强（迁移成本高）
- 已有核心产品，新增功能工程量可控

**劣势：**
- 需要企业级支持能力（SLA、驻场支持、定制开发）
- 销售周期长（一般 1-3 个月决策周期）
- 需要 BD/销售资源，个人开发者难以独立推进

**关键成功因素：** 找到 2-3 家种子企业客户做试点，用实际案例打开市场。

### 2.2 路径 B：Nexus 网络增值服务

**商业模式：** 局域网基础功能免费，广域网高级服务按层级付费

**分层定价：**

| 层级 | 月费 | 能力 |
|------|------|------|
| 免费层 | 0 | 局域网 mDNS 发现 + 5 个联系人 + 基础 A2A 对话 |
| 专业层 | 19 元 | 广域网连接 + 专属信令节点 + 50 联系人 + 对话历史云端加密备份 |
| 团队层 | 99 元 | 私有信令服务器 + 无限联系人 + Agent 发布到公共网络 + 优先匹配 |

**远景——Agent 社交网络：**

这是灵犀最具想象力的方向。想象一个未来场景：

> 你创建了一个精通法律的 Agent，装备了法规知识库和合同审查技能。你把它发布到 Nexus 公共网络。其他用户的 Agent 遇到法律问题时，可以自动发现并调用你的 Agent。每次调用你获得收入。你的 Agent 积累了信誉评分，排名越来越高，调用量持续增长。

这本质上是一个 **Agent 经济体**：
- Agent 有身份（名称、能力标签、所有者）
- Agent 有声誉（调用次数、评分、成功率）
- Agent 有收入（按调用计费，提供方获得 70%）
- Agent 有社交关系（信任关系、协作历史）

**优势：**
- 网络效应一旦形成，壁垒极深（梅特卡夫定律：网络价值 ∝ 节点数²）
- 天花板极高，有可能成为 "Agent 互联网"的基础设施层
- 现有 Nexus 架构（mDNS + 信令 + Transport 抽象层）已打好技术基础

**劣势：**
- 冷启动极难——网络中节点不够时没有价值，没有价值就吸引不来新节点
- 变现周期最长（预计 12-18 个月才能看到有意义的收入）
- 广域网 P2P 面临 NAT 穿越、信令服务器成本、消息可靠性等技术挑战

**关键成功因素：** 冷启动策略（详见第四部分）。

### 2.3 路径 C：Agent/技能/知识库市场

**商业模式：** 平台抽佣 20-30%

**三类市场：**

| 市场 | 内容 | 定价区间 | 基础设施要求 |
|------|------|----------|------------|
| Agent 模板市场 | 精调智能体配置包（prompt + 技能绑定 + 参数） | 9.9-99 元/个 | Agent 导出/导入功能 |
| 技能市场 | 开发者上传的技能包 | 免费-49 元/个 | 已有 Smithery 基础 |
| 知识库市场 | 行业知识库（法规、医学、产品手册等） | 49-499 元/套 | 知识库导出/导入功能 |

**生态飞轮逻辑：**

更多创作者 → 更丰富的内容 → 更多用户 → 更高的交易量 → 更多创作者收入 → 吸引更多创作者 → ...

**优势：**
- 边际成本极低（平台做分发，内容由社区创作）
- 生态飞轮一旦转起来，自我强化
- 丰富产品内容，提升用户留存和使用深度

**劣势：**
- 严重依赖用户量（DAU < 10,000 时市场几乎无流动性）
- 内容质量参差不齐，需要审核机制
- 知识产权纠纷风险（别人的 prompt 被二次售卖等）

**关键成功因素：** 先做好 Agent 导出/导入基础设施，再逐步开放市场。

### 2.4 路径 D：个人订阅制

**商业模式：** 免费版（功能受限） + Pro 版（19 元/月）

**功能分级：**

| 功能 | 免费版 | Pro 版 |
|------|--------|--------|
| 智能体数量 | 3 个 | 无限 |
| 技能数量 | 5 个 | 无限 |
| 知识库数量 | 2 个 | 无限 |
| 工作流节点 | 3 个以内 | 无限 |
| 定时任务 | 不可用 | 完整支持 |
| Nexus 局域网 | 不可用 | 完整支持 |
| Nexus 广域网 | 不可用 | 完整支持 |
| 新功能优先体验 | 无 | 有 |
| 跨设备加密同步 | 无 | 可选附加 |

**收入测算：**

| 场景 | 总用户数 | 转化率 | 付费用户 | 月收入 | 年收入 |
|------|---------|--------|---------|--------|--------|
| 起步期 | 5,000 | 3% | 150 | 2,850 | 34,200 |
| 成长期 | 50,000 | 3% | 1,500 | 28,500 | 342,000 |
| 规模期 | 200,000 | 4% | 8,000 | 152,000 | 1,824,000 |

**优势：**
- 开发成本最低（主要是加 license 校验逻辑，约 2-3 周）
- 用户转化路径短，自助式
- 可与其他路径叠加，不冲突

**劣势：**
- 个人用户付费意愿普遍较低（尤其是国内市场）
- 功能限制过严会影响口碑传播和开源社区评价
- 收入天花板相对有限

**关键成功因素：** 功能分级要精心设计——免费版要够用到让用户"上瘾"，但 Pro 版的增值功能要让重度用户觉得"值"。

### 2.5 路径综合对比与组合策略

**四条路径一览对比：**

| 维度 | 路径 A 企业版 | 路径 B Nexus | 路径 C 市场 | 路径 D 订阅 |
|------|:-----------:|:----------:|:---------:|:---------:|
| 变现速度 | 快 | 慢 | 中 | 中 |
| 收入天花板 | 中高 | 极高 | 高 | 中 |
| 护城河深度 | 深 | 极深 | 中 | 浅 |
| 开发投入 | 大 | 中 | 中 | 小 |
| 用户量依赖 | 低 | 高 | 极高 | 中 |
| 运营复杂度 | 高（需BD） | 中 | 高（需审核） | 低 |
| 风险等级 | 中 | 高 | 中 | 低 |

**推荐组合策略（不建议只走一条路）：**

**第一阶段（0-6 个月）：路径 D + 路径 A 试点**
- 立即上线个人订阅（开发成本最低，快速验证付费意愿）
- 同步开发企业版核心功能（RBAC + 审计日志），找 2-3 家种子企业客户试点
- 此阶段核心指标：付费用户转化率、种子企业客户签约

**第二阶段（6-12 个月）：叠加路径 C**
- 用户量积累到一定规模后开放 Agent 市场
- 引入创作者激励计划，冷启动内容供给
- 此阶段核心指标：市场 GMV、创作者数量、Agent 安装量

**第三阶段（12-24 个月）：推进路径 B**
- 广域网 Nexus 稳定后推出付费信令服务
- 逐步构建 Agent 社交网络生态
- 此阶段核心指标：Nexus 网络节点数、Agent 调用量、网络收入

## 第三部分：功能路线图

### 3.1 功能优先级矩阵

按"商业价值"和"开发难度"两个维度对所有候选功能进行评估：

**第一象限——优先开发（高价值 + 低难度）：**
- Agent 导出/导入包
- Prompt 版本管理
- 审计日志
- 对话模板/蓝图
- RBAC 权限

**第二象限——战略投资（高价值 + 高难度）：**
- Multi-Agent 团队编排
- Agent 市场
- 本地向量 RAG
- 目标分解与长任务

**第三象限——按需开发（低价值 + 低难度）：**
- 图片生成集成
- 数据看板
- 时间胶囊

**第四象限——长期规划（低价值 + 高难度）：**
- 协作白板
- 代码执行沙箱
- Agent 人格进化

### 3.2 Phase 1：商业化基座（0-3 个月）

**目标：** 实现基础变现能力，提升核心产品粘性。

| # | 功能 | 描述 | 工期 | 涉及模块 |
|---|------|------|------|----------|
| 1 | Agent 导出/导入包 | `.lingxi-agent` 格式打包（JSON manifest + prompt + 技能引用 + 参数配置），一键导入/导出 | 1-2 周 | 前端 AgentFactoryPage + 后端 handler/agent.go + 新导出逻辑 |
| 2 | Prompt 版本管理 | agents 表增加 prompt_history JSON 字段，支持版本列表、对比查看、一键回滚 | 1 周 | 后端 db/db.go + handler/agent.go + 前端向导 |
| 3 | License 与付费门控 | 免费版/Pro 版功能分级，前端门控 + 后端校验。支持 License Key 激活 | 2-3 周 | 新增 license 模块（前后端） |
| 4 | RBAC 多用户权限 | 基于现有 users 表扩展 role 字段（admin/member/viewer），Gin 中间件鉴权 | 3-4 周 | db/db.go + handler/auth.go + 中间件 + 前端权限组件 |
| 5 | 审计日志 | 新增 audit_logs 表，记录关键操作（对话/Agent 变更/API Key 操作），支持查询和导出 | 2-3 周 | db/db.go + 新 handler/audit.go + 前端设置页 |
| 6 | 对话模板/蓝图 | 预设对话流程（N 个结构化问题 → 用户逐一回答 → Agent 生成结构化输出），模板 CRUD | 2 周 | 新 conversation_templates 表 + handler + 前端 UI |

**Phase 1 完成后的产品形态：** 灵犀具备基础付费能力（个人 Pro 订阅），企业版最小可用（RBAC + 审计），Agent 可以被导出分享（为后续市场做准备）。

### 3.3 Phase 2：生态构建（3-9 个月）

**目标：** 构建 Agent 市场生态，增强 Agent 自主能力。

| # | 功能 | 描述 | 工期 | 依赖 |
|---|------|------|------|------|
| 1 | Agent 市场 | 浏览/搜索/发布/安装/评分，基于 Phase 1 的 Agent 导出包格式 | 4-6 周 | Agent 导出/导入包 |
| 2 | 工作流运行时引擎 | 当前工作流是 UI 编排，需实现后端执行引擎（节点调度 + 变量传递 + 错误处理 + 执行日志） | 4-6 周 | 无 |
| 3 | 自动化管线 | 定时任务触发工作流 → Agent 执行 → 结果推送 IM，形成端到端自动化链路 | 3-4 周 | 工作流引擎 |
| 4 | 目标分解与长任务 | Agent 接收高级目标 → 自动分解子任务 → 进度追踪 UI → 断点续跑 | 3-4 周 | 无 |
| 5 | 本地向量 RAG | 集成轻量 embedding（onnxruntime 或 llama.cpp 跑 embedding 模型），本地向量索引 + BM25/语义混合检索 | 4-6 周 | 无 |
| 6 | 数据看板 | 对话统计仪表盘、Agent 使用热度排行、模型性能对比、知识库引用热力图 | 2-3 周 | 无 |

**Phase 2 完成后的产品形态：** 灵犀从"AI 聊天工具"升级为"AI 自动化平台"——Agent 能自主分解任务、工作流能真正运行、定时任务能串联 IM 推送、市场能发现和安装社区智能体。

### 3.4 Phase 3：网络效应（9-18 个月）

**目标：** 构建 Agent 网络，实现多 Agent 协作，推进平台化。

| # | 功能 | 描述 | 工期 | 依赖 |
|---|------|------|------|------|
| 1 | Multi-Agent 团队编排 | 一个任务分配给多个 Agent 组成的团队，角色分工 → 并行执行 → 汇总审批 | 6-8 周 | 目标分解 |
| 2 | Agent 声誉系统 | Nexus 网络中 Agent 的能力评分、历史表现、信任等级、调用统计 | 3-4 周 | Agent 市场 |
| 3 | 广域网 Nexus 付费层 | 专属信令节点、高级 P2P 穿越（TURN 中继）、SLA 保障 | 4-6 周 | 现有信令服务器 |
| 4 | 跨设备 P2P 同步 | 利用 Nexus 基础设施实现个人多设备间本地加密同步（会话 + 配置 + Agent） | 4-6 周 | Nexus 广域网 |
| 5 | 代码执行沙箱 | 本地 Docker/WASM 沙箱执行 Agent 生成的代码，文件系统和网络隔离 | 6-8 周 | 无 |
| 6 | Agent 人格进化 | 基于长期记忆自动提炼用户偏好模式，自适应微调 prompt 参数 + 对话风格 | 3-4 周 | 长期记忆系统 |

**Phase 3 完成后的产品形态：** 灵犀成为一个完整的 "Agent 操作系统"——Agent 能组队协作、能跨设备同步、能安全执行代码、能持续进化学习，Nexus 网络开始产生网络效应。

## 第四部分：Nexus 网络效应构建策略

Nexus 是灵犀最独特的技术壁垒，也是最有想象力的商业方向，值得单独讨论其冷启动和增长策略。

### 4.1 网络效应飞轮模型

```
更多灵犀用户 → 更多在线 Agent → 更丰富的 Agent 能力网络
      ↑                                        ↓
吸引更多用户加入 ← 单个 Agent 价值提升 ← Agent 市场交易量 ↑
      ↑                                        ↓
      └──────── 创作者收入增加 ← 更多优质 Agent 上架 ←┘
```

网络效应的关键拐点：当网络中的 Agent 种类覆盖了大部分常见需求（翻译、代码审查、数据分析、文案创作...），新用户加入时就能立即发现有价值的 Agent 可以协作——这时飞轮就开始自旋了。

### 4.2 冷启动三阶段

**阶段一：企业内部网络（局域网）—— 第 0-6 个月**

目标：验证 Agent 协作在真实工作场景中的价值。

策略：
- 面向 10-50 人技术团队推广"团队内 Agent 协作"场景
- 经典用例：代码审查员 Agent + 架构师 Agent + 测试 Agent 在团队内自动协作
- 局域网 mDNS 零配置，使用门槛为零
- 收集用户反馈，打磨核心协作体验

关键指标：
- 活跃局域网集群数
- 平均每集群 Agent 对话次数/周
- 用户满意度（NPS）

**阶段二：跨企业伙伴网络（广域网）—— 第 6-12 个月**

目标：验证跨组织 Agent 协作的商业价值。

策略：
- 合作伙伴间建联场景：甲方的需求分析 Agent 与乙方的方案设计 Agent 自动对话
- 引入企业认证 + 能力认证机制，建立信任基础
- 广域网信令服务分免费/付费两层

关键指标：
- 跨组织建联数
- 广域网对话成功率
- 付费信令用户转化率

**阶段三：公共 Agent 网络 —— 第 12-24 个月**

目标：构建开放的 Agent 经济体。

策略：
- 开放 Agent 注册与公共发现（按能力标签索引）
- Agent 调用计费：提供方获得 70%，平台抽佣 30%
- 声誉系统上线：调用次数 + 评分 + 成功率 → 综合信誉等级
- Agent 推荐算法：根据历史协作记录推荐最适合的 Agent

关键指标：
- 公共网络注册 Agent 数
- 月活跃调用量
- 市场 GMV
- Agent 提供方月均收入

## 第五部分：创新功能构想

以下是一些更具想象力的功能方向，可作为长期储备：

### 5.1 Agent 自主规划与长任务执行

当前 Agent 主要在单轮或简单多轮对话中工作。升级方向：
- 用户给出高级目标（"帮我写一份商业计划书"）
- Agent 自动分解为 5-10 个子任务，生成执行计划
- 每个子任务有状态（待处理/进行中/完成），用户可看到总进度
- 子任务失败自动重试或请求人类介入
- 断点续跑：中断后从上次进度继续

### 5.2 自动化管线——工作流 + 定时 + IM 的终极融合

场景举例：
> 每天早上 9:00，"市场分析 Agent"自动抓取行业新闻 → 生成中文摘要 → 交给"翻译 Agent"翻译成英文 → 自动推送到企业微信群 → 如有重大新闻，触发"报告 Agent"生成详细分析报告推送给领导

这是将定时任务、工作流引擎、Agent 能力、IM 集成串联成一条完整的自动化管线。

### 5.3 事件驱动 Agent（"时间胶囊"）

- 给 Agent 设置未来触发条件："当比特币跌破 5 万美元时通知我"
- Agent 定期通过 MCP 工具检查条件
- 条件满足时主动发起对话通知用户
- 本质：从"被动响应"升级为"主动监控"

### 5.4 Agent 人格持续进化

- Agent 不只是一个固定 prompt，它随着交互不断成长
- 自动从对话中提炼用户偏好（已有长期记忆基础），主动调整回答风格
- Agent 性格迁移：把一个 Agent 的"经验"移植到新 Agent
- "数字分身"：模仿你说话风格的 Agent，可以代你回复常见问题

### 5.5 协作白板

- 在对话旁边打开一个可视化画布
- Agent 的输出可以拖到画布上，用户手动组织和标注
- 支持节点间连线（思维导图 / 流程图模式）
- 最终一键导出为结构化文档
- 本质：AI 参与的可视化知识整理工具

### 5.6 Prompt 版本管理与 A/B 测试

- 每次修改 Agent 的 system prompt 都有完整历史记录
- 支持两个版本并行运行，对比效果差异
- "上周的 prompt 效果更好" → 一键回滚到历史版本
- 为 Agent 市场的内容质量控制奠定基础

### 5.7 多设备 P2P 同步（不过云端）

- 利用 Nexus 基础设施，在用户自己的多台设备间加密同步
- 手机上查看桌面 Agent 的执行进度
- 手机上审批 Agent 的执行结果
- 纯本地 P2P，数据不经过任何第三方

## 第六部分：关键风险与应对

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|----------|
| **大厂入场** — OpenAI/Anthropic 推出原生桌面 Agent 工作台 | 高 | 中 | 强化"模型无关 + 本地化"差异点。大厂不会做"支持所有竞争对手模型"的产品。差异化方向：多模型切换、Nexus 网络、企业数据合规 |
| **用户增长瓶颈** — 桌面应用分发效率低于 Web SaaS | 中 | 高 | 考虑推出 Web 版（复用 Go 后端 + React 前端），降低首次试用门槛。桌面版作为全功能版保留 |
| **开源同质化** — LobeChat、AnythingLLM 等项目功能追赶 | 中 | 中 | Nexus 网络 + 工作流引擎 + 企业版功能组合形成多层壁垒。单一功能容易被抄，但完整生态很难被复制 |
| **模型价格战** — 调用费用持续下降，削弱成本管理卖点 | 低 | 高 | 价值锚点从"省钱"转向"Agent 自动化效率提升"和"数据安全合规"。模型降价反而有利于提升 Agent 使用频率 |
| **技术债** — 功能快速迭代积累的代码质量问题 | 中 | 中 | 在 Phase 1 和 Phase 2 之间安排专门的代码质量冲刺（测试覆盖、性能优化、架构重构） |
| **合规风险** — AI 监管政策变化影响产品模式 | 中 | 低 | 本地化架构天然规避大部分数据合规问题。持续关注政策动态，保持架构灵活性 |

## 第七部分：下一步行动建议

### 如果现在就开始，推荐执行顺序：

**第一优先级（立即开始）：**

1. **Agent 导出/导入包**（1-2 周）
   - 投入最小，但为后续 Agent 市场奠定基础
   - 用户可以分享智能体给朋友/同事，天然传播渠道

2. **Prompt 版本管理**（1 周）
   - 开发量极小（agents 表加一个 JSON 字段 + 前端版本列表 UI）
   - 用户强需求，大幅提升产品粘性

**第二优先级（第 2-4 周）：**

3. **License 门控 + 付费页面**（2-3 周）
   - 开始产生收入，验证付费意愿
   - 对现有功能进行分级，不需要新功能

**第三优先级（第 4-8 周）：**

4. **RBAC + 审计日志**（3-4 周）
   - 企业版最小可用版本
   - 可以开始找种子企业客户

**第四优先级（第 8-14 周）：**

5. **工作流执行引擎**（4-6 周）
   - 让工作流从"画图"变成"真正能跑"
   - 解锁自动化管线场景

### 阶段里程碑：

| 时间节点 | 里程碑 | 核心指标 |
|----------|--------|----------|
| 第 2 周 | Agent 导出/导入 + Prompt 版本管理上线 | 导出/导入次数 |
| 第 5 周 | 个人 Pro 订阅上线 | 首月付费转化率 |
| 第 9 周 | 企业版 MVP（RBAC + 审计）上线 | 种子客户签约数 |
| 第 15 周 | 工作流引擎上线 | 工作流创建/执行次数 |
| 第 6 个月 | Agent 市场 Beta 版上线 | Agent 上架数量 + 安装量 |
| 第 12 个月 | Nexus 广域网稳定 + 付费层上线 | 广域网节点数 + 付费转化率 |
